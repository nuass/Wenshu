# Technical — 技术主张

## 核心判断：教育 AI 的关键不是模型，是记忆

大多数教育 AI 产品把力气花在"更好的大模型"上。
我们的判断不同：**模型够用了，缺的是上下文。**

一个没有记忆的 AI 家教，每次都从零开始——它不知道你上周错了什么，
不知道你对哪个概念反复困惑，不知道你的进步在哪里。
再强的模型，没有上下文，给出的也是通用建议。

---

## 我们的技术架构

### 1. 持久化教学记忆层

```
答题行为
   ↓
结构化记忆文件（Markdown + frontmatter）
   ├── 学生画像：学习习惯、偏好、目标
   ├── 教学反馈：答题规律、薄弱点、进步信号
   └── 学习进展：章节掌握情况、里程碑
   ↓
memory_summary 注入 LLM prompt
   ↓
带有"记忆"的个性化回复
```

**为什么不用向量数据库：**
教学记忆是结构化的、可解释的，不是语义检索问题。
Markdown 文件对老师透明可读，对 AI 直接可用，部署零依赖。

### 2. PDF → 题库 Pipeline

```
PDF 扫描题目
   ↓
OCR + LLM 解析（题干、选项、答案、解析、知识点标签）
   ↓
结构化 JSON 题库
   ↓
智能选题引擎（薄弱点权重 + 难度自适应 + 去重窗口）
   ↓
飞书互动卡片推送
```

### 3. 选题引擎的评分逻辑

每道题的推送分数由以下因素决定：
- 知识点掌握度（越薄弱权重越高）
- 难度匹配（自适应当前学生水平）
- 去重窗口（N 天内不重复推同一题）
- 错题优先（答错过的题权重提升）

### 4. 飞书 Cardkit 状态卡片

用飞书 Cardkit 卡片实体实现答题后实时变色：
- 选正确答案 → 按钮变绿
- 选错误答案 → 按钮变红，同时展示解析
- 支持多题批量提交（`1A2B3C` / `答案是ABD`）

## 技术选型原则

| 决策 | 选择 | 理由 |
|------|------|------|
| LLM 接入 | OpenAI 兼容接口 | 可换任意模型，不锁定供应商 |
| 记忆存储 | Markdown 文件 | 零依赖，人类可读，git 友好 |
| 飞书通信 | lark-cli | 屏蔽 token 管理，专注业务逻辑 |
| 后台框架 | Flask + Alpine.js | 轻量，单文件部署 |
| 定时任务 | 系统 crontab | 可靠，无额外依赖 |

---

## 护城河不在模型，在数据飞轮

每一次学生答题 → 记忆更准确 → 下次推题更精准 → 学生效果更好 → 更多答题数据

这个飞轮一旦转起来，积累的**学生学习档案**就是最深的护城河。
